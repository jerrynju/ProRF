# Pro Workflow App Matrix 统一界面设计语言

## 1. 文档定位

本文档定义一套面向移动端原生工程应用矩阵的统一界面设计语言。

目标不是把桌面端大型画布软件缩小到手机屏幕里，而是把大型工程工作站拆解为一组聚焦任务的独立 App。每个 App 解决不同问题，但共享同一套对象模型、导航结构、交互方式、视觉语言和实现约束。

本文档分为三部分：

1. 指导原则：定义产品哲学和判断标准。
2. 通用落地方案：所有 App 共同遵守，必须使用通用名称，不出现具体领域词。
3. 射频链路计算适配：说明这套通用语言在射频链路计算 App 中如何落地。

## 2. 指导原则

### 2.1 移动端原生，而不是桌面端折叠

移动端 App 不应是桌面端软件的压缩版。

必须遵守：

- 启动后进入有用状态，而不是空白大画布。
- 优先使用模板、推荐和引导创建，而不是要求用户从零搭建。
- 优先使用流程式编辑，而不是自由画布编辑。
- 优先使用底部面板和详情页，而不是桌面式左右面板。
- 专家能力必须存在，但不能成为默认首屏体验。

### 2.2 结果优先

用户打开 App 时，首先应理解当前设计的结果、风险和下一步动作。

必须遵守：

- 主页面顶部展示当前状态。
- 关键结果、风险提示和建议动作优先于底层细节。
- 图表、摘要和解释是一等界面，不是运行后的附属弹窗。
- 原始日志、执行轨迹、底层数据放入高级入口。

### 2.3 App 矩阵，而不是超级 App

产品体系应拆成一组独立 App。每个 App 聚焦一个主要任务，但共享同一套设计语言。

必须遵守：

- 每个 App 有清晰边界和主要工作流。
- 相同对象在所有 App 中使用相同名称、状态、布局和手势。
- 用户学会一个 App 后，应能自然理解其他 App。
- 共享组件不能写入具体领域假设。

### 2.4 渐进式复杂度

界面必须分层暴露复杂能力。

标准分层：

1. 引导层：模板、推荐、自动连接、一键运行、自然语言解释。
2. 编辑层：参数编辑、组件替换、场景对比、简化结构编辑。
3. 专家层：自由画布、类型化连接、监看、单步、断点、原始数据、自定义组件。

默认体验必须落在引导层或编辑层。专家层只能作为可进入的高级能力。

### 2.5 结构共享，含义插件化

所有 App 使用同一套结构词汇：

- 项目
- 设计
- 组件
- 连接
- 参数
- 场景
- 结果
- 问题
- 库

具体领域含义由各 App 通过 schema、模板、校验规则、计算器和结果渲染器注入。

### 2.6 先解释，再暴露细节

系统反馈必须被翻译成用户能理解和行动的信息。

问题反馈必须包含：

- 发生了什么。
- 为什么重要。
- 影响哪个组件或设置。
- 用户下一步可以做什么。

错误码、原始校验信息和执行日志可以存在，但不能作为默认表达。

### 2.7 Schema 驱动 UI

可复用 UI 应尽量由 schema 生成。

共享 schema 包括：

- 组件 schema
- 参数 schema
- 连接 schema
- 场景 schema
- 结果 schema
- 问题 schema

UI 层只读取 schema 和状态，不包含领域公式或计算逻辑。

## 3. 通用落地方案

本章是所有 App 共同遵守的通用规范。除“领域适配”章节外，共享文档、共享导航、共享组件和共享 API 必须使用本章中的通用名称，不使用具体领域词。

### 3.1 App 矩阵架构

每个 App 都遵循同一结构：

```text
App
  项目
    设计
      组件
      连接
      分组
    场景
    运行记录
    结果
    问题
    库引用
```

### 3.2 共享对象模型

| 对象 | 含义 | UI 角色 |
|---|---|---|
| 项目 | 一组工作的容器 | 首页卡片、最近打开、设置 |
| 设计 | 可运行的结构 | 主编辑对象 |
| 组件 | 可复用的操作单元或元素 | 流程行、卡片、详情页 |
| 连接 | 组件之间的关系 | 流程连接、画布连线 |
| 参数 | 可配置值 | 表单、滑杆、步进器、单位输入 |
| 场景 | 一组命名参数值 | 对比卡片、运行设置 |
| 运行记录 | 一次执行记录 | 历史、状态、复现 |
| 结果 | 运行输出 | 仪表盘、图表、报告 |
| 问题 | 校验或运行反馈 | 问题卡片、修复动作 |
| 库条目 | 可复用组件或模板 | 添加面板、搜索、集合 |

这些名称是所有 App 的共同界面语言。领域 App 可以在内容区展示领域名称，但共享导航、共享组件、共享文档和共享接口必须使用通用名称。

### 3.3 主导航模型

所有 App 默认使用同一主导航：

```text
设计 / 运行 / 结果 / 库 / 更多
```

轻量 App 可以收敛为：

```text
设计 / 结果 / 库
```

导航规则：

- 设计是默认工作区。
- 结果是一等入口，不是弹窗。
- 库用于添加组件、模板和可复用分组。
- 运行承载运行设置、场景、批量执行和历史。
- 更多承载项目设置、导入导出、账户、帮助和高级工具。

### 3.4 标准页面范式

#### 3.4.1 项目首页

作用：展示最近工作和当前状态。

必须包含：

- 最近项目
- 固定模板
- 最近运行状态
- 未处理问题
- 继续操作

首页不能是空白编辑器。

#### 3.4.2 设计总览

作用：在编辑细节前概览当前设计。

必须包含：

- 设计名称和状态
- 关键结果卡片
- 问题数量
- 主操作：运行或查看问题
- 次操作：添加组件或编辑设计

#### 3.4.3 流程视图

作用：移动端默认编辑视图。

流程视图把设计表达为有序、可阅读、可点击的结构。它比自由画布更适合手机上的浏览、编辑和复查。

必须支持：

- 点击组件进入详情。
- 点击连接查看相关数据。
- 点击插入点添加组件。
- 在结构允许时长按调整顺序。
- 通过滑动或菜单执行复制、替换、禁用、删除。

#### 3.4.4 画布视图

作用：专家级结构编辑。

当用户需要非线性布局、显式连接、分组或高级检查时，进入画布视图。

规则：

- 默认自动布局。
- 手动定位是可选能力。
- 创建连接必须支持“选择源，再选择目标”，不能只依赖拖拽。
- 高级手势必须有可见替代入口。
- 画布不能是创建有效设计的唯一方式。

#### 3.4.5 组件详情

作用：单个组件的唯一编辑和检查入口。

默认结构：

```text
摘要
参数
输入
输出
图表
问题
高级
```

规则：

- 摘要优先。
- 常用参数直接可见。
- 高级参数默认折叠。
- 输入和输出根据 schema 自适应渲染。
- 问题必须包含建议动作。

#### 3.4.6 运行设置

作用：配置并启动执行。

必须包含：

- 当前场景
- 运行选项
- 执行范围
- 校验状态
- 开始操作

运行设置必须在不打开原始配置的情况下可理解。

#### 3.4.7 结果总览

作用：解释运行后发生了什么。

必须包含：

- 总体状态
- 关键结果卡片
- 图表
- 场景对比
- 问题和建议
- 导出或分享动作

结果必须先可读，再允许用户查看原始表格。

#### 3.4.8 问题中心

作用：聚合校验和执行反馈。

问题卡结构：

```text
严重程度
标题
影响对象
自然语言解释
建议动作
跳转动作
```

问题不是日志。日志属于高级工具。

### 3.5 组件卡片体系

所有 App 使用三种组件卡密度。

#### 紧凑卡

用于列表和小摘要。

```text
组件名称
主值 · 次值
```

#### 标准卡

用于流程视图。

```text
组件名称
类型或角色
关键参数
主输出或状态
```

#### 展开卡

用于详情页和平板布局。

```text
组件名称
状态
参数
输入
输出
问题
```

规则：

- 标准卡最多展示三个关键值。
- 优先展示角色、状态和结果，不优先展示底层 schema 字段。
- 所有 App 共享同一状态模型。

### 3.6 共享状态

| 状态 | 含义 |
|---|---|
| 未配置 | 缺少必要信息 |
| 就绪 | 可以运行 |
| 运行中 | 正在执行 |
| 已完成 | 最近一次结果有效 |
| 需更新 | 参数或输入在最近一次运行后发生变化 |
| 警告 | 可用但存在风险 |
| 错误 | 无法产生有效结果 |
| 已禁用 | 不参与执行 |

各 App 内部状态必须映射到这些共享状态。

### 3.7 参数输入规则

参数字段使用统一结构：

```text
标签
数值输入 + 单位
简短说明
必要时显示校验信息
```

规则：

- 带单位的值是一等输入。
- 有推荐范围时应直接显示。
- 有界调节使用滑杆。
- 小步离散调节使用步进器。
- 精确值和表达式使用文本输入。
- 高级表达式必须作为可选能力。

### 3.8 库规则

库用于添加组件、模板和可复用分组。

必须包含：

- 搜索
- 最近使用
- 推荐
- 分类
- 模板
- 自定义条目

规则：

- 从库添加组件不能只依赖拖拽。
- 每个库条目必须有自然语言说明。
- 模板必须创建可运行起点，而不是静态示例。
- 推荐条目可以基于当前设计上下文。

### 3.9 场景规则

场景是一组命名参数值。

必须支持：

- 从当前设计创建。
- 复制并编辑。
- 对比结果变化。
- 标记基准。
- 运行一个或多个场景。

场景对比应优先展示结果变化和问题变化，而不是原始参数差异。

### 3.10 结果规则

结果不是原始输出数据，而是主要解释层。

结果层级：

1. 总体状态
2. 关键值
3. 趋势或图表
4. 解释
5. 建议动作
6. 原始数据

原始数据必须可访问，但不是第一层。

### 3.11 高级工具

高级工具在所有 App 中共享，但隐藏在默认路径之后。

可包含：

- 自由画布编辑
- 类型化连接
- 监看
- 单步执行
- 断点
- 原始执行日志
- Schema 查看器
- 导入导出
- 自定义组件编辑器

规则：

- 高级工具不能阻塞基础任务完成。
- 高级入口应解释用户为什么可能需要它。
- 高级页面可以更高密度，但仍遵守共享对象模型。

## 4. 共享界面语言

### 4.1 命名规范

共享 UI 和共享文档必须使用通用名称。

| 使用 | 避免在共享 UI 中使用 |
|---|---|
| 设计 | 图、图纸、模型专有名称 |
| 组件 | 块、节点、器件专有名称 |
| 连接 | 线、导线、链路专有名称 |
| 详情 | 检查器 |
| 问题 | 诊断控制台 |
| 结果 | 数据显示器 |
| 运行设置 | 求解器设置 |
| 分组 | 子系统专有名称 |
| 监看 | 探针 |
| 运行记录 | 执行日志 |

领域名称只能出现在领域 App 的内容区、模板、组件名称、参数名称、结果名称中，不能进入共享组件 API、共享导航和共享设计系统。

### 4.2 动作词

共享主动作：

- 创建
- 添加
- 运行
- 查看
- 对比
- 替换
- 调整
- 导出
- 分享

共享高级动作：

- 监看
- 单步
- 暂停
- 继续
- 校验
- 查看原始数据

### 4.3 视觉气质

界面应现代、清晰、可信，而不是桌面工程软件的菜单堆叠。

规则：

- 默认使用浅色背景和白色内容面。
- 不把密集桌面面板作为默认布局。
- 不使用装饰性背景。
- 不使用强领域主题化视觉。
- 颜色用于状态、分类和图表识别。
- 图标帮助扫描，但不能替代必要文字。
- 触控目标不小于 48dp。

### 4.4 色彩系统

共享语义色：

| 角色 | 颜色 |
|---|---|
| 主色 | #2563EB |
| 成功 | #16A34A |
| 警告 | #F59E0B |
| 错误 | #DC2626 |
| 信息 | #2563EB |
| 需更新 | #8B5CF6 |
| 禁用 | #98A2B3 |

中性色：

| 角色 | 颜色 |
|---|---|
| 背景 | #F7F8FA |
| 内容面 | #FFFFFF |
| 主文本 | #1D2433 |
| 次文本 | #667085 |
| 边框 | #D9DEE7 |

分类色由具体 App 提供。共享 UI 只消费 `category.primary`、`category.secondary`、`category.accent` 等分类 token。

### 4.5 字体层级

推荐移动端字号：

| 角色 | 字号 |
|---|---|
| 页面标题 | 24sp |
| 分区标题 | 18sp |
| 卡片标题 | 16sp |
| 正文 | 14sp |
| 元信息 | 12sp |
| 关键值 | 28sp 到 36sp |

规则：

- 关键值可以大。
- 单位应比数值弱。
- 不按屏幕宽度动态缩小文字。
- 优先缩短文案，而不是压缩字号。

## 5. 实现约束

### 5.1 分层架构

```text
领域插件
  Schemas
  Templates
  Validators
  Calculators
  Result Renderers

共享 App Shell
  Navigation
  Design View
  Flow View
  Canvas View
  Details
  Run
  Results
  Library
  Issues

共享 UI Components
  Cards
  Forms
  Charts
  Sheets
  Lists
  Navigation
```

共享 UI 不能依赖领域插件。领域插件依赖共享 UI 合约。

### 5.2 强制隔离

共享 UI 禁止包含：

- 领域公式
- 硬编码领域单位假设
- 领域组件分类
- 领域校验规则
- 领域图表定义

共享 UI 可以包含：

- 带单位的通用输入组件
- 通用图表容器
- 通用结果卡
- 通用问题卡
- 通用分类色 token
- 通用 schema 渲染器

### 5.3 Schema 合约

最小组件 schema：

```text
id
name
role
category
parameters[]
inputs[]
outputs[]
display[]
advanced[]
```

最小结果 schema：

```text
status
key_values[]
charts[]
issues[]
raw_data[]
suggested_actions[]
```

最小问题 schema：

```text
severity
title
affected_object
explanation
suggested_actions[]
jump_target
```

## 6. 领域适配：射频链路计算 App

本章说明通用界面语言在射频链路计算领域的具体落地。领域词只允许出现在本章或具体产品实现中，不能反向污染通用规范。

### 6.1 产品定位

射频链路计算 App 是一个移动端原生的链路设计、计算和分析工具。

它不应像一个缩小版桌面仿真器，而应帮助用户快速回答：

- 当前链路结果是什么。
- 哪一级对结果影响最大。
- 哪个余量存在风险。
- 下一步应该调整哪个参数。
- 不同场景之间有什么差异。

### 6.2 默认用户流程

推荐流程：

```text
选择模板
  → 查看链路
  → 编辑关键参数
  → 运行
  → 查看结果
  → 修复问题或对比场景
```

默认编辑界面是流程视图。画布视图提供给需要显式拓扑编辑的用户。

### 6.3 射频链路对象映射

| 通用对象 | 射频链路计算中的含义 |
|---|---|
| 项目 | 一个设计研究或产品上下文 |
| 设计 | 一条射频链路或一个链路预算模型 |
| 组件 | 信号源、增益级、损耗级、路径、天线、接收机、测量点或自定义元素 |
| 连接 | 链路元素之间的信号流 |
| 参数 | 增益、损耗、频率、功率、噪声系数、带宽、距离、温度等射频量 |
| 场景 | 工作条件、环境、频段、温度条件或架构选项 |
| 结果 | 链路增益、输出功率、接收功率、噪声、余量、灵敏度或图表 |
| 问题 | 范围越界、缺少参数、单位不匹配、压缩风险、余量不足或链路无效 |
| 库条目 | 可复用链路元素、模板或计算模式 |

### 6.4 射频设计总览

必须包含：

- 链路状态
- 关键射频结果卡片
- 风险或余量摘要
- 最近运行时间
- 主操作：运行、查看问题、添加元素

推荐关键卡：

- 总增益或总损耗
- 输出功率或接收功率
- 噪声相关指标
- 余量
- 主要限制元素

总览必须解释结果。例如，当余量较低时，应说明哪个元素或场景导致风险。

### 6.5 射频流程视图

射频流程视图把链路表达为可阅读的纵向序列。

每个元素行应展示：

- 元素名称
- 元素角色
- 两到三个关键射频值
- 当前输出或贡献
- 状态

规则：

- 有序链路应便于纵向扫描。
- 元素之间显示插入入口。
- 用户应能不进入完整画布就替换元素。
- 所有数值必须带单位。
- 默认不展示所有参数。

### 6.6 射频组件详情

推荐页签：

```text
摘要
参数
输入
输出
图表
问题
高级
```

射频参数行为：

- 所有工程量使用带单位输入。
- 已知推荐范围时直接展示。
- 派生值标记为只读。
- 详细公式放入高级或帮助。
- 警告用余量、有效性或物理含义解释。

### 6.7 射频结果总览

结果层级：

1. 总体通过、警告或错误状态。
2. 关键射频值。
3. 链路贡献图。
4. 场景对比。
5. 自然语言解释。
6. 参数或元素调整建议。
7. 原始值和导出。

推荐图表：

- 沿链路的功率或电平变化。
- 增益和损耗贡献。
- 噪声贡献。
- 各场景余量。
- 灵敏度或门限对比。

### 6.8 射频问题

射频问题卡必须遵守通用问题格式。

示例：

```text
标题：输出电平接近限制
解释：某个元素接近推荐工作范围，可能降低可用余量。
建议动作：降低上游增益、加入衰减，或选择线性度更高的元素。
```

```text
标题：缺少必要参数
解释：某个元素没有定义必要参数，因此无法完成计算。
建议动作：打开该元素详情并填写缺失值。
```

规则：

- 不只显示公式错误。
- 必须指出受影响元素。
- 尽可能提供至少一个可执行建议。

### 6.9 射频模板

模板必须创建可运行起点，而不是静态示例。

推荐模板类型：

- 简单链路
- 接收链路
- 发射链路
- 链路预算
- 测量配置
- 空白设计

每个模板应包含：

- 简短说明
- 适用场景
- 默认参数场景
- 关键结果卡
- 已知假设

### 6.10 射频高级模式

射频高级工具可包含：

- 自由画布拓扑
- 显式类型化连接
- 信号监看
- 参数扫描
- 原始结果表
- 导入导出
- 自定义元素定义

高级模式不能取代默认的流程优先体验。

## 7. 迁移建议

推荐实施顺序：

1. 将共享 UI 概念统一改为通用对象模型。
2. 将流程视图设为移动端默认设计界面。
3. 将结果提升为主导航入口。
4. 将底层执行细节移入高级入口。
5. 将组件详情改为共享详情结构。
6. 将射频分类色和领域标签迁移到领域插件数据中。
7. 增加带解释和建议动作的问题卡。
8. 保留画布视图作为带自动布局的专家模式。

## 8. 总结

这套统一界面设计语言的核心是：

```text
移动端原生
结果优先
流程优先
Schema 驱动
渐进式专家能力
默认领域中立
通过插件注入领域语义
```

射频链路计算 App 应使用这套语言成为一个友好的工程助手：启动快、结果清楚、问题可行动，同时在需要时仍然具备专家级图结构编辑能力。
